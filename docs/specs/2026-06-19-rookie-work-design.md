# rookie-work 设计 spec

> 一个面向 agent 新手的「工作质量兜底」Claude Code 插件。
> 状态：设计定稿（2026-06-19），待写实现计划。

## 0. 一句话

让 agent 在帮新手干活时，**凡是要动手、或要替用户拍板的事，都先用听得懂的话讲清楚、并让用户做选择**——给不懂「该向 agent 要什么纪律」的新手兜住工作质量。

## 1. 背景与受众

- **受众**：刚开始用 agent、还不知道「该让 agent 怎么工作」的新手。
- **痛点**：新手不知道该要求 agent 先规划、先确认；agent 又容易**静默**做掉一堆有后果的决定（直接改代码、自选模型、自行其是）。出了问题，新手也看不懂错在哪。
- **目标**：把「久经考验的工作纪律」用**大白话**默认套在 agent 身上，让新手不必懂行，也能得到稳的过程与结果。

## 2. 核心承诺与三原则

**招牌承诺（写进 README 和技能开场）：**

> agent 绝不会在没用你听得懂的话讲清「要做什么、怎么做」、并得到你点头之前，改动你的任何东西。

**三条原则：**

1. **先交底再动手** — 任何实际修改前，用通俗语言讲清「做什么 + 怎么做」，确认方向才动。
2. **不静默拍板** — 每个需要拍板的边界/默认，都摆成「带利弊 + 推荐」的选择题交给用户。
3. **量体裁衣** — 三档分流，简单任务不被流程拖累。

## 3. 与 superpowers 的关系

- **借鉴思想，不照搬文本**：先设计后写码、给多方案 + 推荐、先计划后实现等，都是通用工程纪律、非 superpowers 私有；我们用更简单、为新手调过的话**重写**。
- **零运行时依赖**：用户无需知道或安装 superpowers。
- **多做两样 superpowers 没有的**：① 向用户复述项目轮廓并**确认 agent 真的懂了**；② 把每个**静默默认摆成选择题**（模型、是否评审等）。

## 4. 形态与分发

- **形态**：一个 **Claude Code 插件**，内含：
  - `rookie-work` **技能**（`SKILL.md`，指令用英文写、要求 agent 用「用户的语言」回应）；
  - 一个 **SessionStart 钩子**，实现「默认常开」；
  - 一份 **README**（讲清它做什么，让人放心安装）。
- **分发**：推上 GitHub，并提供一个**插件市场仓库**，新手 `/plugin install` 一键装、可更新。
- **语言**：跟随用户语言（指令英文，交流用用户语言）。
- **待核准**：插件清单 / 钩子 / 市场仓库的**精确文件格式**，在写实现计划/动手时对照 Claude Code 官方文档核准（必要时派 `claude-code-guide` 代理确认）。

## 5. 启用与关闭

- **默认常开**：SessionStart 钩子在每次会话开始注入 rookie-work 的工作约定，使 agent 开工即遵循。
- **可关**：
  - **口头当场关**：用户说「关掉新手模式 / 给我原始模式」→ 本会话停用；
  - **持久关**：钩子检查一个标记位（如某标记文件/设置项），让「关掉」跨会话保留；
  - 也支持**显式调用**重新启用。
- 注：这是整个项目唯一的「harness / 钩子」成分；其余全是技能本体。

## 6. 三档分流（决定走多重的流程）

| 档 | 何时 | 怎么做 |
|---|---|---|
| **0 · 直接做** | 只读 / 查询 / 定位 / 解释，**不改任何东西** | 零仪式，直接给结果 |
| **1 · 轻量** | 小改动、低风险、**做法唯一且明显** | 交底（做什么 + 怎么做）→ 确认方向 → 做 + 报告 → 越界前停 |
| **2 · 全套** | 有设计空间 / 影响面大或不清 / **难撤销** | 走完整 8 步 |

- **硬线**：只读 → 直接做；一旦「写 / 改 / 删 / 装 / 发布 / 提交」→ **最低 Tier 1，永不静默跳过**。
- **透明 + 可推翻**：凡涉及改动，agent 一句话亮明「归到第几档、为什么」；用户随时**一键升档 / 降档**。
- **防自我开脱**：禁止 agent 把该认真对待的事自我合理化成「这很简单」来跳过流程——所以「改动必亮档」是硬性的。

## 7. 全套 8 步（Tier 2）

1. **吃透项目** — 读项目（含 §10 的变更日志）→ 用大白话向用户复述轮廓与关键点 → **问「我这样理解对吗」等确认**。
2. **复述需求 + 摆边界** — 复述要做的事 → 挑出需要拍板的边界 → 逐个摆成「利弊 + 推荐」选择题。
3. **定方案再写码** — 给 2-3 个方案 + 推荐理由；**达成一致前绝不碰代码**。
4. **派子代理时（若用到）** — 摆出模型策略（见 §11）。
5. **先计划后实现** — 把认可的方案落成轻量小步骤计划 → 用户过目。
6. **实现** — 照计划做；**中途要超范围 / 改已定方案，先停下征同意**。
7. **（可选）评审** — 问是否要多一道评审，客观讲利弊由用户定（见 §12）。
8. **验收** — **有据才算完成**（真验证，不是嘴上说好了）；遗留项记入 backlog 不丢。

## 8. Tier 1（轻量）流程

① 我要做 X → ② 用大白话讲清「做什么 + 怎么做」、确认方向是否正确、有没有要补充说明的 → ③ 做 + 报告 → ④ 越界前停。

## 9. 「动手前先交底」铁律（贯穿 Tier 1 & 2）

任何**实际修改**动手前，都必须用初学者能懂的通俗语言讲清「要做什么 + 怎么做」，确认方向后才动。唯一豁免是 **Tier 0（纯只读）**。对 Tier 2，这条已被第 2/3/5 步天然涵盖；对 Tier 1，即 §8 的 ②。

## 10. 留痕 / 变更记录

- **Tier 0**：不落文件。
- **Tier 1 + Tier 2**：都向一份**追加式变更日志**（`CHANGELOG`）写一条：**改了什么 / 为什么 / 何时 / 影响范围 / 哪一档**。一行一条，不新建文件。
- **Tier 2 额外**：再落一份**大白话短计划 / 设计**文档，并从变更日志条目链接过去。
- **闭环**：第 1 步「吃透项目」会**读这份变更日志**了解改动历史——记录喂回给未来的 agent。
- **位置**：默认 `docs/rookie-work/`（`CHANGELOG.md` + 大改的计划文档），可配置。

## 11. 子代理模型策略（静默默认摆成选择题之一）

将要启用子代理时，向用户摆出两种策略：

- **A · 设默认**：用户选一个默认模型；之后子代理都用它，**除非本次显式指定**别的。
- **B · 每次问**：每派一个子代理都让用户挑模型。

用户随时可显式「换策略」重选；**agent 主动备注**这个口子存在。利弊讲大白话（省 token、简单任务不必上最强模型）。**不预设任何「默认用最强模型」的个人偏好**。

## 12. 评审策略（静默默认摆成选择题之二）

多轮 / 跨模型评审**不静默执行**：先**征求确认**，用新手能懂的话**客观讲利弊**（多一道保险 vs 多花 token / 时间），并**允许用户说出自己的想法**（不只是 yes / no）。

## 13. 贯穿性元原则：静默默认 → 讲明白的选择

凡是**会花钱 / 花时间 / 增加复杂度、或 agent 本会自己拍板**的事，都不静默执行，而是用新手能懂的话摆成「带利弊 + 推荐」的选择题。§6 的分档、§11 的模型、§12 的评审，都是它的实例。

## 14. 非目标（YAGNI）

- 不做 superpowers 面向高手的全部重型能力；只给新手兜底。
- 不在琐碎任务上强加重型文档（三档分流已规避）。
- 不做自动重试 / 自动退避等高级编排。
- 不替用户做它该拍板的任何决定。

## 15. 待核准 / 待定（写计划时解决）

- 插件清单 / SessionStart 钩子 / 市场仓库的**精确文件格式**（对照官方文档）。
- 持久「关闭」标记的具体机制（标记文件 vs 设置项）。
- 变更日志默认位置与格式细节。
- 三档分类的判定提示如何写得让 agent 稳定执行（防「自我开脱」）。
- 本 spec 是否需出英文版供公开仓库（现为中文）。

## 16. 成功标准

新手装上后：

- 琐碎请求**依旧快**（Tier 0 零摩擦）；
- 任何改动**动手前必有大白话交底**并经确认；
- 该拍板的边界**都以选择题形式呈现**（带推荐）；
- 每次改动**有可追溯记录**，大改有文档；
- 用户**随时能升降档、能整体关掉**。
