# rookie-work 更新设计：Tier 2 的「框架化 → 边界发现」前置阶段

> 状态：设计已与用户敲定（2026-06-20），待写实现计划。
> 关联：基础设计见 `2026-06-19-rookie-work-design.md`；多 agent 分发见 `2026-06-19-多agent支持-design.md`。

## 1. 背景与问题

rookie-work 的理念之一是 **「用户应该了解自己在 vibecoding 什么，每一步决策之后的结果会是什么样、是否符合自己想要的」**。但现实里很多用户卡在**最开始那一步**就过不去：

- **场景 1（冷启动）**：大量普通 vibecoding 用户只有自己专业领域的知识、或只有"想要的最终效果"，**不具备系统开发知识，提不出一个能跟 AI 一起头脑风暴、不断完善、不断确定边界的开发框架**。而框架恰恰重要——它约束了 AI 产出偏离用户想法的可能，也保障用户始终"看得懂自己在做什么"。
  - 用户本人就是其中一员：在 ship-platform 的输入模式重构与数据库搭建中，因缺乏系统开发知识而提不出可共同开始的框架。其现有解法是在简单需求后追加一句后缀「请你理解我的上述需求，并将上述语言转化成便于 AI 理解工作的 prompt」，让 AI 把需求扩写成带专业开发视角的 prompt。
- **场景 2（多轮边界发现）**：重要工程需要用户与 AI 反复头脑风暴、不断确定边界；用户缺乏开发知识，需要 AI **从开发/专业角度主动提问**；用户提出的内容常**牵连到 ta 不知道的其它边界/设计层面**，需要 AI 替 ta 揭示；而且重要工程**一轮谈不完**，需要 AI 主动确认"用户是否已理解当前已定的边界、还有没有别的边界要定"，**主动帮用户发现并确定边界**。

### 1.1 对现有后缀法的评估（用户明确要求）

用户现有方案【原始需求 +「请理解并转成便于 AI 工作的 prompt」】：

- **可行性：有效、零成本、今天可用，核心直觉正确**——把"外行话翻译成开发语言"外包给 AI。
- **结构性短板（对照 rookie-work 理念）**：
  1. **一次性，不成环**：扩写成一段 prompt 就结束，没建立"逐条边界、反复细化"的框架；用户拿到更高级的 prompt，却未必更懂其中边界。
  2. **产出物是「给 AI 的 prompt」，不是「用户能懂的共识」**：专业 prompt 恰是外行最看不懂的东西，易照单全收，把 AI 填入的假设当成自己的决定——**正踩"不静默做决定"的反面**。
  3. **AI 扩写时偷偷补假设**：模糊需求→专业 prompt 的过程必然脑补大量边界；只输出结果 prompt 时，这些脑补被"藏"进去，用户无机会逐个确认。
  4. **依赖用户记得这个技巧**：真正的外行不知道有此后缀；而 rookie-work 是 **agent 侧**护栏，本应由 agent 主动发起。
- **结论**：保留后缀法的好内核（让 AI 做翻译），但升级为 **agent 主动发起、对话式、逐边界确认**的过程，产出"用户真正理解并拍板的开发框架"。

### 1.2 关键洞察：两个场景是同一件事的两端

- 场景 1 = **冷启动**：把"提不出框架"变成一个**初始框架**。
- 场景 2 = **之后的多轮循环**：AI 主动从开发视角提问、挖出用户不知道的隐藏边界、每轮确认"懂了吗/还有要定的吗"。

二者合一 = **给外行用户的、AI 主导的「框架化 → 边界发现」前置阶段**，挂在 **Tier 2 的前半段**，作为单一连贯功能设计。

## 2. 目标与非目标

**目标**
- 让接不住框架的用户，也能在 Tier 2 动设计之前，得到一个**自己看得懂、亲自拍板**的开发框架。
- 让 AI **可靠地**（不靠当场灵感）替外行揭示其不知道的隐藏边界。
- 多轮收敛：主动确认理解、主动追问"还有什么要定"，直到框架够清楚再进设计。
- 给一份**可复用、可移植**的框架化提示模板（即使没装 rookie-work 也能用）。

**非目标（YAGNI）**
- 不新增档位、不新建独立 skill；这是 Tier 2 前半段的加强，复用现有"一套方法"。
- 不替用户做技术选型决定——只把边界**摊成大白话选择题**让用户拍板（守"不静默"）。
- 不强制每个 Tier 2 都跑框架化——仅在需求对外行而言模糊时**提议**。

## 3. 与核心理念的契合

- **先解释再动手**：框架化全在"动代码前"完成，产出共识后才进设计。
- **不静默做决定**：边界以"选项+利弊+推荐+理由"呈现，用户拍板；杜绝把 AI 脑补当用户决定。
- **右尺度**：只在检测到模糊时提议；清晰需求直接走正常 Tier 2，不打扰。
- **用户理解自己在 vibecoding 什么**：产出是"用户能懂的框架共识"，不是给 AI 的 prompt——直接兑现该理念。

## 4. 设计总览（落点）

这是**把现有 Tier 2 第 2 步「复述任务 + 摊开决策」升级加强**，并在第 0 步加一个触发判断。现有 Tier 2 八步骨架不变，仅前半段对"接不住框架的用户"变厚。理由：保住"一套方法"的连贯性，并让三家（CC/Codex/Hermes）经现有 `build/build.sh` 自动同步。

## 5. 组成部分（五件）

### 5.1 触发：检测 → 主动提议（已定）
Agent 判为 Tier 2 且需求对外行而言**框架不清/欠定义**时，主动提议：「这是个大任务、框架还没定，要不要我先陪你把框架搭出来？」——**用户点头才进**；已给清晰框架的直接走正常流程。是**提议不是强加**（守"不静默"）。

### 5.2 冷启动框架化（对话式，非一次性扩写）（已定）
Agent 把用户的模糊需求**翻成一个大白话的初始框架**（"我理解你要的是…、这活大致长这样"），让用户确认/纠正。**产出是用户能懂的共识**，不是给 AI 的 prompt——即后缀法的升级版。

### 5.3 内置「边界维度清单」：后台扫，只拎相关的问（已定）
Agent 拿下面这份清单当**后台扫描表**，逐项过一遍，但**只把"跟该任务相关、且需用户拍板"的拎成大白话选择题**（带选项 + 利弊 + 推荐 + 理由）——**不照单逐项念**（避免变成填问卷/审讯）：

1. **范围与非目标** — 要做什么；明确**不**做什么
2. **谁用、怎么用** — 使用者、典型使用场景
3. **输入 → 输出** — 喂进去什么、吐出来什么
4. **数据形态与留存** — 数据长什么样、存哪、要不要留历史
5. **规模量级** — 几条还是百万条、几人用、多频繁
6. **出错怎么办** — 异常 / 空数据 / 用户乱输入时的行为
7. **集成与依赖** — 要不要连外部系统 / 账号 / 第三方服务
8. **运行环境** — 本地 / 网页 / 手机 / 服务器……
9. **完成判据** — 什么算"做完"、怎么验证、给谁看
10. **硬约束** — 预算、时间、隐私 / 合规、绝对不能动的东西

清单是**可裁剪的起点**，非穷举；Agent 按任务相关性增删。

### 5.4 多轮收敛循环（一轮定不完）（场景 2 核心）
每轮结束，Agent：
- (a) 用大白话**复述已定的边界**；
- (b) **确认用户真的懂了**（不是光点头）；
- (c) 问"**还有要定的吗**"；
- (d) **主动指出清单里仍未定的维度**。

循环直到「用户表示够了 **且** Agent 没有遗留的关键维度」，才进入 Tier 2 第 3 步（设计）。

### 5.5 可复用的「框架化提示模板」（便携）（已定：与主线并存）
把用户后缀法打磨成正式版，做成一段**可粘贴**提示词，放进新参考文档；用户即使在**没装 rookie-work** 的 AI 里也能贴上使用。模板正文（拟）：

```
这是我的大致需求：
【在这里写你的需求，有多简单写多简单】

我可能不太懂开发，请先别直接开做。先帮我把它变成一个我能看懂的开发框架：
就下面这些方面，一项一项问我（每次只问一项、用大白话、给我选项和你的推荐、并说明为什么推荐；
不要替我默默决定）：
范围与非目标 / 谁用怎么用 / 输入输出 / 数据与留存 / 规模量级 /
出错怎么办 / 集成与依赖 / 运行环境 / 完成判据 / 硬约束。
每问完一轮，帮我复述一遍已经定下来的边界、确认我真的听懂了，再问我还有没有别的要定的。
等框架定得差不多了，再进入具体设计。
```

## 9. 决策记录（本次头脑风暴）

- **冷启动机制** = **两者都要**：Agent 主动框架化对话（主）+ 可复用提示模板（便携）。
- **触发时机** = **检测到需求太模糊时主动提议**（用户点头才进；清晰需求不打扰）。
- **边界发现** = **内置「边界维度清单」后台扫、只拎相关且需拍板的问**（既系统又不像审讯）。

## 10. 范围

**一份 spec + 一份实现计划**即可，无需拆分：五个部分围绕同一前置阶段，改动集中在 `SKILL.md` / 一个新参考 / `SESSION-PREAMBLE.md` / `plugin.json` / `README.md` / `tests/`。

## 11. 风险与待实测

- **触发判定的尺度**：何谓"对外行模糊"由 Agent 判断，可能偏松或偏紧——靠"提议而非强加 + 用户可一句话跳过"兜底；真机冒烟观察。
- **清单不照念的执行度**：5.3 要求"后台扫、只拎相关"，但模型可能仍倾向逐项问——SKILL.md 文案需明确"背景扫描、按相关性收敛、不逐项念"。
- **多轮循环的收敛**：避免无限追问——5.4 收敛判据写清"用户够了 且 无遗留关键维度"即停。
- **preamble 体积**：常开层只加一句，细节留 SKILL.md，避免拖累每会话注入成本。
