# rookie-work 行为可靠性升级设计

> 状态：设计已由用户逐项确认（2026-07-10）；实现与独立审查修复已完成。用户决定本轮不执行三平台真机验证，当前实现未做真机验证。
> 范围：只升级行为可靠性；发布、版本调整与全项目质量升级不在本轮自动执行。

## 1. 背景

rookie-work 已经具备三档分流、Tier 2 八步流程、模糊需求框架化和边界发现能力。实际会话表明，这套方法能够形成用户喜欢的工作节奏：先回顾已确认边界，再提出下一项选择，说明真实方案、利弊和推荐，最后等待用户决定。

现有规则仍有三个可靠性缺口：

- “真正选择”缺少按结果影响判断的完整定义，Agent 仍可能把重要产品决定当成技术细节。
- 确认节奏、漏答、决定冲突和会话恢复等行为没有形成统一协议。
- 现有测试以文件结构、关键文字和打包一致性为主，不能证明 Agent 在真实对话中稳定执行这些规则。

本次升级的目标，是把已经验证有效的对话方式固化成明确、可解释、可测试的行为协议，同时保持现有用户习惯和三平台结构兼容。

## 2. 已确认的产品边界

### 2.1 本轮范围

- 聚焦行为可靠性升级。
- 保留 Tier 0、Tier 1、Tier 2 三档。
- 保留 Tier 2 的八步流程。
- 保留当前启用、关闭和三平台安装方式。
- 同时覆盖 Claude Code、Codex 和 Hermes，继续使用同一套规则源。
- 完成本轮后，再判断是否有必要安全地启动全项目质量升级。

### 2.2 什么是真正选择

尚未确定的事项只要会改变以下任一结果，就属于必须交给用户确认的真正选择：

- 用户实际看到或得到的产品行为；
- 业务规则、成功或失败口径；
- 本轮范围与明确不做的内容；
- 数据保存、隐私、安全或权限；
- 成本、时间或外部依赖；
- 兼容性、可逆性或风险。

用户已经明确决定的内容只需复述和记录，不重复询问。不会改变上述结果的纯技术细节，可以由 Agent 在批准范围内处理。Agent 无法可靠判断时，必须询问而不是自行归类。

### 2.3 与任务分档的关系

- Tier 0 仍是纯只读任务。
- Tier 1 只能用于低风险、且做法唯一明显的小改动。
- Tier 1 在执行前或执行中一旦发现真正选择，必须停止并升级为 Tier 2。
- Tier 2 使用本设计规定的完整选择协议。

### 2.4 方案质量

每个真正选择应提供 2–3 个真实可行方案，并说明：

- 选择后实际会发生什么；
- 主要优点；
- 主要缺点或代价；
- Agent 推荐哪一个以及原因。

不能为了凑数量编造陪衬方案。客观上只有一个安全、合法或可行方案时，Agent 应说明为什么其他方向不可行，再让用户决定继续、修改目标或暂停。用户始终可以提出自己的方案。

## 3. 选定的设计路线

本轮采用“行为协议为主，示例与场景为辅”的混合方案。

曾考虑的路线包括：

1. 分层行为协议：规则清晰、可维护，但单独使用时缺少足够的对话示例。
2. 全部规则内联：Agent 能直接看到全部内容，但会显著增加常驻上下文和重复维护成本。
3. 示例与测试驱动：直观、改动小，但示例不能代替通用判断原则。

最终方案以分层行为协议作为正式规则，再使用正确示例、反例、固定场景和真实 Agent 冒烟帮助理解与验证。硬原则是：**规则优先；示例用于解释规则，但不能覆盖、缩小或替代规则。**

## 4. 整体结构与文件职责

### 4.1 常驻提醒层：`SESSION-PREAMBLE.md`

只放每次会话都必须立刻生效的硬规则：

- 发现真正选择时不能静默决定；
- Tier 1 发现真正选择就升级为 Tier 2；
- Tier 2 开始边界确认前先询问用户的确认节奏，并告知可以随时切换；
- 沉默、漏答或模糊表达不能视为同意。

这里不放完整示例，以控制常驻上下文长度。

### 4.2 正式流程层：`SKILL.md`

保留现有三档和八步骨架，在对应步骤中接入：

- 真正选择的结果影响判定；
- 确认节奏贯穿整个 Tier 2；
- 方案呈现要求；
- 即时确认与阶段回顾；
- 冲突、越界和上下文不完整时的处理；
- 分级完成状态。

`SKILL.md` 是正式流程入口，并指向行为协议参考文件。

### 4.3 行为协议层：新增 `references/decision-protocol.md`

集中保存：

- 真正选择与纯技术细节的定义；
- 单项和批量两种确认节奏；
- 节奏切换；
- 批量漏答与歧义；
- 安全、可行性和旧决定冲突；
- 即时确认、阶段回顾和决定覆盖；
- “实现完成”与“完整验证完成”的区别；
- 正确示例与反例。

所有示例必须注明其非穷举性质。

### 4.4 分发层

- 根目录继续作为唯一真实来源。
- 现有 `build/build.sh` 继续生成 Codex 和 Hermes 包。
- Claude Code 继续使用根目录插件结构。
- 不增加平台专属行为分支。
- 生成后的三个平台必须包含一致的行为协议。

规则优先级为：安全与平台硬规则 → 用户明确决定 → `SKILL.md` 正式规则 → 行为协议细节 → 示例。

## 5. 真正选择的处理流程

### 5.1 分档与进入 Tier 2

Agent 先进行任务分档。Tier 0 直接执行只读工作；Tier 1 先交底并征得同意。Tier 1 一旦出现真正选择，立即停止并升级为 Tier 2。

Tier 2 先理解项目并向用户复述。用户确认理解正确后，才进入边界确认。

### 5.2 先让用户选择确认节奏

在提出第一个边界前，Agent 让用户选择：

1. 严格一次只确认一个选择；
2. 每次集中确认两到四个相关选择。

Agent 同时说明：

- 该节奏覆盖边界、设计、计划、实施中途新选择和可选评审等整个 Tier 2 任务；
- 用户可以在任何时候切换；
- 切换从下一轮开始生效，不推翻已经确认的内容。

### 5.3 后台识别与前台呈现

Agent 在后台扫描真正选择，只把需要用户决定的内容摆到前台。用户已明确决定的内容直接记录，不重复询问；纯技术细节不进入选择清单。

每个选择使用统一的方案质量要求。若采用单项节奏，每轮只处理一个未定选择；若采用批量节奏，每轮只组合两到四个相互关联的选择。

### 5.4 只记录明确答案

- 只记录用户明确表达的决定。
- 漏答、含糊或互相冲突的内容保持未确定。
- 不把沉默、简短附和或未回应视为接受推荐。
- 批量模式下，复述已经明确的答案，再继续询问未确定项。
- 用户主动一次确认多个事项时可以全部记录，不机械重问。

### 5.5 任务内共识

每项决定分为：`已确认 / 未确定 / 明确推迟`。

- 每次选择后即时复述新决定。
- 一个自然阶段结束时集中回顾全部已定、未定和推迟事项。
- 用户修改旧决定时，以新决定明确覆盖旧决定，并说明受影响的后续内容。
- 只有用户认为框架足够清楚，并且 Agent 没发现重要未定项时，才能进入下一阶段。

## 6. 冲突、异常与越界处理

### 6.1 不推荐但安全可行

Agent 说明风险与推荐理由。用户明确坚持后，以用户选择为准，不继续劝阻，也不暗中改回推荐方案。

### 6.2 不安全、违规或技术上不可行

Agent 不能假装接受。必须说明具体冲突、判断依据、仍可保留的目标和安全可行的替代方案。用户可以调整目标、选择替代方案或暂停。

### 6.3 与旧决定冲突

Agent 列出原决定、新决定、二者差异和受影响内容。用户明确选择后才更新共识；不能同时实施互相冲突的新旧决定。

### 6.4 实施中发现新选择

Agent 在执行受影响的修改前停止。可以继续与该选择无关的只读检查，但不能继续会预设答案的实现工作。用户可以推迟选择，但依赖该选择的工作也必须同步暂停。

### 6.5 无回复、会话恢复和上下文不完整

- 用户未回复时维持未确定状态，不自动采用推荐方案。
- 会话恢复或压缩后，优先从当前记录、设计文档和变更记录恢复共识。
- 无法可靠确认旧决定时，明确指出缺失信息并重新询问，不能凭印象补齐。

### 6.6 工具或环境受限

Agent 区分代码问题与环境无法验证。可以完成不依赖该环境的工作，但必须如实记录验证缺口。

## 7. 测试与验收

### 7.1 第一层：确定性规则与打包测试

扩展现有 Shell 测试，检查：

- `SESSION-PREAMBLE.md` 包含四条常驻硬规则；
- `SKILL.md` 包含完整选择流程并正确引用行为协议；
- 行为协议文件存在并包含全部核心定义与示例声明；
- Claude Code、Codex 和 Hermes 得到一致规则；
- 关闭开关、三档流程和现有构建行为没有退化；
- 构建可重复，重新生成后不存在意外漂移。

这一层只证明规则正确写入并分发，不宣称 Agent 已实际遵守。

### 7.2 第二层：固定行为场景

建立可读的场景清单。每个场景包含：

- 场景编号与目的；
- 用户输入和当前任务状态；
- Agent 必须采取的行为；
- Agent 明确禁止采取的行为；
- 属于自动检查、真实冒烟，还是两者都要。

核心场景至少覆盖：

- Tier 1 发现真正选择并升级；
- 用户已经明确决定时不重复询问；
- 单项模式与批量模式；
- 中途切换节奏；
- 批量漏答和模糊回答；
- 不编造假方案；
- 只有一个可行方案；
- 用户修改旧决定；
- 安全可行但不推荐；
- 不安全、违规或不可行；
- 实施中发现新选择；
- 会话恢复后无法确认旧决定；
- 平台无法真机验证；
- 发布不自动发生。

每条通用规则至少配一个正确示例和一个反例。

### 7.3 第三层：真实 Agent 冒烟

在每个实际可用平台的新会话中运行代表性关键场景：

- 代表场景包括 Tier 1 升级、确认节奏与切换、批量漏答、实施中新选择、安全或旧决定冲突；
- 每个代表场景必须连续通过两次；
- 任何失败都要记录，不能只保留成功结果；
- 修正规则后重新开始连续通过计数；
- 记录平台、Agent 或模型信息、日期、输入、输出摘要和判断依据；
- 无法运行的平台标记为“待真机验证”。

### 7.4 完成状态

- `实现完成`：规则、文档、固定场景、构建与一致性检查全部通过。
- `平台真机验证完成`：对应平台的代表场景连续通过两次。
- `三端完整验证完成`：三个平台都完成真机验证。
- 发布不自动发生，需要另行确认。

## 10. 设计风险与控制

- **常驻上下文膨胀**：前置文件只保留四条硬规则，详细内容放入技能和参考层。
- **规则与示例漂移**：明确规则优先级，测试要求每条通用规则同时拥有正确示例和反例。
- **模型只模仿格式**：固定场景覆盖未见过的表述，真实冒烟验证行为而不是关键词。
- **三平台行为分叉**：继续使用单一来源和确定性构建，验证生成内容一致。
- **验证结果被夸大**：使用分级完成状态，未真机运行的平台始终保留待验证标记。
- **范围膨胀**：安装、品牌、发布和全项目质量升级都明确排除在本轮实现之外。

## 11. 成功标准

本轮设计在实施后应做到：

- Agent 能按结果影响识别真正选择；
- 用户能在 Tier 2 开始时选择确认节奏，并随时切换；
- 漏答、沉默和模糊表达不会被当成同意；
- Tier 1 不再成为绕过重要决策的入口；
- 冲突、安全问题、实施中新选择和上下文缺失都有一致处理；
- 规则、示例、固定场景和真实 Agent 冒烟形成可追踪证据链；
- 三平台保持同源，同时如实区分实现完成与真机验证完成。
